# Search-friendly authoring guide

This guide explains how to make Verifik docs **easier to find** by users who type
country aliases, regional terms, or raw API paths in the search bar. It pairs
with the on-site search engine described in `internal/SEARCH_BACKEND.md`.

There are three knobs, ordered from cheapest to most explicit:

1. **Front-matter `keywords`** — terms emitted to `<meta name="keywords">` and
   used by the local search index for ranking.
2. **`<SearchKeywords>` MDX component** — visible chips at the bottom of the
   page that expand a central country/term map (`internal/search-synonyms.json`).
3. **Generated reference page** — every endpoint URL is auto-listed in
   `docs/reference/endpoint-doc-index.md`, regenerated from
   `internal/docs-i18n-inventory.json`.

## 1. Front-matter `keywords`

Add a flat YAML list of strings. Keep it short and human; do not stuff.

```yaml
---
id: epic-eci
title: India EPIC (Voter ID) — Person & Name Lookup | Verifik
slug: /legal/epic-eci
keywords:
  - India
  - EPIC
  - Voter ID
  - ECI
  - electoral verification
  - KYC
---
```

Docusaurus emits these into the page `<head>` automatically. The local search
plugin includes them in the document boost.

## 2. `<SearchKeywords>` component

For high-value pages, render a visible "Also known as" block. It pulls aliases
from `internal/search-synonyms.json` so a single source of truth covers every
page that mentions the same country or term.

```mdx
import SearchKeywords from '@site/src/components/SearchKeywords';

# India — EPIC (Voter ID) name lookup

...page content...

## About Verifik

...

<SearchKeywords
  countries={['IN']}
  products={['KYC']}
  documents={['EPIC']}
  apiPaths={['v2/in/epic']}
  extra={['voter id lookup', 'electoral identity']}
/>
```

Available props:

| Prop        | Type       | Looked up in                    | Notes                                   |
| ----------- | ---------- | ------------------------------- | --------------------------------------- |
| `countries` | `string[]` | `synonyms.countries[].code`     | ISO codes; expanded to names + aliases  |
| `products`  | `string[]` | `synonyms.products[].canonical` | KYC, KYB, biometrics, …                 |
| `documents` | `string[]` | `synonyms.documents[].canonical`| cedula, RUES, EPIC, RUNT, …             |
| `apiPaths`  | `string[]` | none — rendered as-is           | Use the literal API path                |
| `extra`     | `string[]` | none — rendered as-is           | Free-form aliases the user might type   |
| `heading`   | `string`   | none                            | Override the localized "Also known as"  |

The block is **visible on purpose** — users get a real cross-reference, and
search indexing comes for free.

### When to add a country / product / document to the central map

Open `internal/search-synonyms.json` and append to the right array. Keep
`canonical` stable; everything else is for matching only.

```json
{
  "canonical": "RUT",
  "countries": ["CL"],
  "aliases": ["Rol Único Tributario", "tax ID", "Chilean tax id"]
}
```

After editing, rebuild:

```shell
npm run build
```

## 3. Generated endpoint → doc index

`docs/reference/endpoint-doc-index.md` is regenerated by
[`scripts/generate-endpoint-doc-index.js`](../scripts/generate-endpoint-doc-index.js).

```shell
npm run inventory-docs-i18n        # refresh inventory JSON
node scripts/generate-endpoint-doc-index.js
```

This runs automatically as part of `prebuild`, so a normal `npm run build`
keeps it up to date. The page is wired into `sidebars.js`.

## 4. URL aliases in front-matter `keywords`

### The slash-tokenization quirk

Lunr's default tokenizer splits on whitespace and hyphens but **not on
slashes**. A path like `v2/co/cedula` is stored as one token, so `cedula` or
`/api/co/cedula` will never hit it through normal indexing. The local search
plugin reads front-matter `keywords` as a separate, title-boosted document
type ([`scanDocuments.js:63-71`](../node_modules/@easyops-cn/docusaurus-search-local/dist/server/server/utils/scanDocuments.js)),
so we surface URL fragments by listing them there.

### One-shot refresh after editing endpoint docs

After adding or renaming any doc with an `### Endpoint` block:

```shell
npm run inventory-docs-i18n         # rescan docs/ for ### Endpoint blocks
npm run inject-endpoint-keywords    # merge URL aliases into each doc's keywords
npm run generate-endpoint-doc-index # rewrite the reference page
```

`npm run inject-endpoint-keywords` is **idempotent** — running it twice does
nothing on the second pass. It also **preserves any manual `keywords`** you
have already curated, only adding the URL alias variants.

CI sanity check (no writes):

```shell
npm run check-endpoint-keywords     # exits non-zero if any doc would change
```

### Aliases produced for one path

For `v2/co/cedula` the injector adds (sample):

```yaml
keywords:
  - v2/co/cedula
  - /v2/co/cedula
  - v2 co cedula
  - api/co/cedula
  - /api/co/cedula
  - api co cedula
  - api.verifik.co/v2/co/cedula
  - 'https://api.verifik.co/v2/co/cedula'
  - GET v2/co/cedula
  - GET /v2/co/cedula
```

The whitespace-tokenized variants make `cedula`, `co`, or `api co cedula`
match. The full-URL variants match copy-paste of the literal endpoint URL.
The `api/...` variants catch a common user mistake (typing `api` instead of
the actual `v2` version segment).

## Anti-patterns

- Do **not** hide keywords with `display:none` or `visibility:hidden`. Users
  benefit from seeing the aliases, and Lunr does not penalize visible terms.
- Do **not** repeat the same word 20 times. The index already boosts titles
  and headings; keyword stuffing only hurts UX.
- Do **not** add per-page synonyms inline if the term is reusable. Add it to
  `internal/search-synonyms.json` so other pages benefit.
